## 핵심 깨달음

### 1. 측정 vs 처리 분리
- `e2e_ms` = GPU 처리만 (capture 빠짐, ~10-14ms)
- `true_e2e_ms` = 카메라 timestamp → GPU 완료 (~18-94ms)
- HARD_LIMIT은 `true_e2e_ms` 기준이어야 의미 있음 (capture 대기 포함된 진짜 데이터 나이)

### 2. 같은 코드 ≠ 같은 결과
- 09:07 (UTC): d1c325f → 79Hz, p50=10.3ms, SOFT 0.49%
- 11:13 (UTC): d1c325f → 39Hz, p50=18.7ms, SOFT 35%
- 같은 code/하드웨어/사람이지만 GPU/ZED SDK의 비결정적 위상 정렬 차이로 결과 변동
- **"원복으로 79Hz 복구"는 운에 맡기는 것 — 코드로 보장 불가**

### 3. 진짜 병목 확정
| 스테이지 | 측정값 | 정상 | gap | 원인 |
|---|---|---|---|---|
| `getdata_rgb` | 11-19ms | 1-2ms | +10ms | RGB-first 순서 → ZED depth sync fence |
| `inf` | 7-15ms | 4ms | +3-11ms | CUDA graph 잡혀도 SM 경합 (ZED depth와 동시) |
| `post` | 5-12ms | 2ms | +3-10ms | 원인 미파악 (.item() 동기화 / NMS / 기타) |

총 합: capture+pipeline = 잠재적 +20ms 절감 가능.

## Plan v2 — 단계별 진행

### Phase 0: Baseline 확정 (완료)
**현재 commit `e9c1844`**: d1c325f + 측정 정직화.
- HARD_LIMIT 20ms 활성화 (true_e2e_ms 기준)
- t_end를 po.stream.synchronize() 직후로 이동
- constraint_ms 별도 측정

### Phase 1: 구조적 fix 적용 (다음 단계)
**작업**: `81db3f0`의 zed_gpu_bridge.py만 cherry-pick (depth-before-RGB 순서)

```bash
git checkout 81db3f0 -- src/perception/CUDA_Stream/zed_gpu_bridge.py
# 또는 수동으로 retrieve_measure를 retrieve_image 앞으로 이동
git commit -m "perf(capture): retrieve depth before RGB (independent of set_stream)"
git push
```

**기대치**:
- `getdata_rgb`: 11-19ms → 1-2ms (10ms 절감)
- e2e p50: 18.7ms → 9-10ms
- FPS: 39 → 60+ Hz
- `true_e2e_ms` p99: 93ms → 25-30ms

### Phase 2: 변동성 측정 (위상 운 정량화)
**작업**: 같은 코드로 60초 × 5회 측정.

```bash
for i in 1 2 3 4 5; do
    sudo ~/realtime-vision-control/src/perception/CUDA_Stream/launch_clean.sh 60 \
        2>&1 | tee run_$i.log
    sleep 30  # GPU 식히기
done
```

**보고 양식**:
- 각 run의 p50, p95, p99, max
- 5 run 평균과 std
- HARD LIMIT 위반율 분포

→ 이게 "오늘 이 Jetson의 d1c325f+depth-before-RGB 진짜 성능 분포".

### Phase 3: 남은 병목 추적 (Phase 1으로 부족할 시)
**3-A. post 스테이지 5-12ms 원인**
- 후보 1: constraint의 `.item()` 호출 (D2H 동기화)
- 후보 2: NMS false positive 처리
- 후보 3: 깊이 패치 lookup 비효율
- 검증: torch.profiler / nsys로 추적

**3-B. inf baseline 7ms 원인**
- CUDA graph capture는 성공
- graph replay 자체는 ~1ms 내 (트레이스 가능)
- 4-7ms gap = SM 경합인지 다른 launch overhead인지

### Phase 4: 추가 최적화 (Phase 3 결론에 따라)
가능한 방향:
- Constraint를 C++ pybind11로 이식 (post .item() 제거)
- CUDA graph 영역을 pre+inf까지 확장
- ZED SDK 호출 패턴 조정

## 영구 기각 (반복 금지)

| 시도 | 이유 |
|---|---|
| TRT INT8 quantization (YOLO26s) | 동작 불량 + keypoint 정확도 손실 (CLAUDE.md에 박힘) |
| `torch.cuda.stream()` → `set_stream()` | context manager의 implicit serialization 제거 → SM 경합 노출 |
| Depth decimation | ZED 내부에서 depth는 매번 계산. retrieve 생략은 sync fence만 누적 |
| `--max-capture-fps` throttle | time.sleep으로 ZED rhythm 깨짐. 위상 정렬 망가짐 |
| 코드 원복으로 특정 FPS 복구 | GPU 시스템 비결정성 → 운에 맡기는 행위. 구조적 fix만 의미 있음 |
