## Codex 5개 질문 답변 핵심 (xhigh reasoning, 응답 archive: `/tmp/codexresp.iMQV8os`)

### Q1 — ZED SDK grab() 큐 동작
- **grab()은 buffer의 latest available frame 반환** (oldest queue가 아님 — 다행).
- **ZED X Mini SVGA 120fps의 정상 GMSL2 latency = 17-25ms** (Stereolabs ROS2 frequency tuning docs).
- ts_ns 의미: 광자가 닿은 *노출 시각이 아님*. GMSL2 deserializer buffer에 *fully available 시점*. 따라서 zed_lag가 17-25ms이면 *정상*, 30+이면 buffer 적체 의심.

### Q2 — Phase A consume-once 영향 (★결정적 통찰)
- **1ms sleep polling은 zed_lag에 직접 영향 없음** (zed_lag는 bridge thread 내부에서 측정됨).
- **본질적 가능성**: Phase A 전에는 *같은 frame_id를 pipeline이 여러 번 받아 처리*했을 가능성. 그 경우 같은 ts_ns가 percentile 분포에서 반복 sampling됨 → 평균이 *낮아 보임*.
- Phase A 후에는 unique frame 기준 → 진짜 분포가 드러남.
- → **8fcfbcc 측정 결과를 그대로 main과 비교하면 unfair**. unique-frame 모드 필수.

### Q3 — ready_event 타이밍
- 현재 ready_ns = H2D enqueue 직후. 그대로 두는 게 맞음 (bridge에 stream.synchronize() 추가 *금지* — 더 느려짐).
- 대신 timestamp *추가* 권장: `h2d_enqueue_ns`, `deque_append_ns`, `pickup_ns` 분리.
- queue_wait를 (host queue) + (h2d 완료 대기) 두 가지로 분해 가능해짐.
- → Phase 0.1로 instrumentation 보강 (측정 후).

### Q4 — Bridge thread 분리
- T1 (capture) / T2 (transfer) 분리 효과: **1-3ms gain**으로 한정 (의미 미미).
- ZED SDK 호출 자체 (grab + retrieve + get_data)는 T1에 남음 → bridge cycle 26ms의 본질 안 풀림.
- **압도적 본질 해법: ZED CUDA interop** — host get_data/D2H/H2D 제거. 단 구현 리스크 큼.
- → Phase B0 (큐 드레인) 우선순위 격하. Phase D (interop) 직진 검토.

### Q5 — HARD LIMIT gate metric (★안전성 구멍)
- 현재 LATENCY_HARD_LIMIT_MS = 20.0이 true_e2e_ms 기준 → 낙관적 (D2H + SHM 추가 시간 무시).
- **올바른 구현**: SHM publish *직전*에 `pre_publish_e2e_ms` 계산해 valid 결정. 현재 main은 이미 이 패턴 적용됨 (run_stream_demo.py:501).
- **빠진 것**: C++ reader가 SHM 읽을 때 *독립* age gate. `if now - ts_ns > 20ms: treat invalid`. Python publish 시점과 C++ read 시점 사이 추가 lag 있을 수 있음.
- → Plan v6.3 (C++ 보강).

## 거시 점검 — Hardware loop 정합성

```
ZED X Mini 120fps (8.3ms 주기)
    ↓ GMSL2 (17-25ms 정상 latency)
[Bridge 26ms cycle ⚠️ — ZED 주기보다 느림 → buffer 누적]
    ↓ deque(maxlen=2)
[Pipeline ~14-20ms cycle] → POSIX SHM seqlock
    ↓
[C++ 100Hz read] → impedance → Teensy → AK60 (max 70N clamp)
    + watchdog 0.2s → pretension 5N fallback
```

**구조적 문제**:
1. Bridge cycle 26ms > ZED 주기 8.3ms → buffer 적체 불가피 (Codex Q1 정상 17-25ms 위에 추가)
2. C++ 100Hz vs perception 67-73Hz → C++가 *같은 frame 두 번* read하는 frame이 30%
3. HARD LIMIT 20ms = true_e2e 기준 → publish 후 추가 lag 무시

**hardware-perception mismatch 처리 방향**:
- Perception 분포 안정화가 우선 (Plan v6.2)
- C++ 측 age gate 추가가 최후 안전망 (Plan v6.3)

### Phase B0 → 우선순위 격하 (Codex Q1)

ZED SDK가 latest를 주는 듯하니, "큐 드레인" 자체는 의미 적음. SDK 내부 buffer는 우리가 못 건드림. 대신:
- **bridge-only 실험** — pipeline 떼고 grab+ts만 60s 측정해 *pure ZED latency 분리*
- 결과가 17-25ms 근처면 정상, 30+이면 grab cadence가 못 따라가는 것 → bridge가 빨라져야

### Phase B1 — post-stage .cpu() D2H sync 제거 (pipeline_proc 우세 시)

`gpu_postprocess.py:266` 의 `.cpu()` D2H sync → keypoint 결과를 GPU에 그대로 두고 D2H를 publish 직전 batch로 모음 (run_stream_demo.py:484-490가 이미 그렇게 함). post 자체에 .cpu() 다시 들어 있으면 제거.

### Phase D — ZED CUDA interop (★ROI 가장 큼, 리스크 큼)

ZED `sl.Mat`을 GPU memory pointer로 직접 받아 → host get_data/D2H/H2D 모두 제거.
- C++ extension에서 PyTorch CUDA primary context 공유
- RGB/depth GPU pointer lifetime + stream ordering 명시

예상 효과: bridge_proc p99 26→8ms, zed_lag p99 43→25ms (정상 GMSL2 latency).

---

### Plan v6.3 — C++ age gate 추가 (Codex Q5)

C++ reader 측에 독립 age gate:
```cpp
auto age_ms = (now_epoch_ns() - shm.ts_ns) / 1e6;
if (age_ms > 20.0) { mark_invalid(); fallback_to_pretension(); }
```

이게 진짜 마지막 안전망. Python publish 시점에 19ms였는데 C++ read 시점에 22ms면 stale data가 control loop에 들어감. 현재 watchdog 0.2s는 너무 관대.

C++ 코드 위치: 사용자에게 확인 필요 (이 repo 또는 별도 control repo).
